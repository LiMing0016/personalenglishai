# 后端技术栈

后端主要技术栈：

- Spring Boot
- MyBatis
- MySQL
- Redis
- REST API
- Maven

后端还包含以下关键基础设施：

- 异步评分任务
- 多引擎语法检查
- 多模型 AI 客户端
- Prompt 组装与上下文管理

## 核心职责

后端主要负责：

- 英语作文评分
- 语法分析
- 写作润色
- 翻译功能
- 题目审核与题目解析
- 写作模板与素材生成
- 用户能力画像
- Prompt 构建
- AI 服务调用
- 评分任务状态管理

## 编码原则

在修改后端代码时，应遵循以下原则：

1. 优先进行 **小范围、安全的修改**
2. 除非明确要求，不要重写大型模块
3. 尽量保持现有 API 接口稳定
4. 保持 `controller -> service -> mapper -> dto/entity/config` 的分层结构
5. 避免在 controller 中写业务逻辑
6. 不要为了“顺手优化”改动不相关链路
7. 修改高风险逻辑时，先说明影响面，再动代码
8. 引入新的框架扩展、Starter、插件或中间件前，先判断现有技术栈是否已经能解决问题

## 框架扩展规则

在考虑新增后端框架扩展、Starter、插件、SDK 或中间件前，必须先评估：

1. 现有 Spring Boot / MyBatis / Redis / Maven 能否直接解决问题
2. 新扩展解决的是缺功能、稳定性问题，还是仅仅为了写法偏好
3. 对配置复杂度、部署方式、日志排障、监控、测试的影响
4. 对现有 API、DTO、数据库、任务状态流转的兼容性

如果决定引入新扩展，必须说明：

- 为什么现有方案不够
- 为什么选择这个扩展而不是在现有链路上小改
- 是否会引入新的运行时依赖、配置项或部署要求
- 是否需要同步更新文档与环境配置

## 分层与架构规范

典型结构：

- controller
- service
- mapper
- entity
- dto
- config

规范说明：

Controller
- 只负责请求处理、参数校验入口、响应返回
- 不承载复杂业务逻辑
- 不直接编排多步 AI 调用、缓存策略或任务状态流转

Service
- 负责业务逻辑
- 负责评分、语法、润色、翻译、能力画像等核心流程
- 负责异步任务、Redis、外部 AI/语法服务的组织与降级

Mapper
- 负责数据库操作
- 保持 SQL 与业务逻辑解耦

DTO
- 用于 API 请求与响应的数据结构
- 保持字段语义稳定，优先向后兼容

Config
- 负责第三方客户端、Redis、Web、安全及系统配置

## AI 模块规则

后端包含大量 AI 相关逻辑，例如：

- 作文评分
- Prompt 构建
- 用户能力画像更新
- 上下文管理
- 模板/素材/润色/翻译生成

重要原则：

1. 不要将评分系统简化为普通文本生成。
2. 需要保持 Prompt 的分层结构：
   - System
   - Role
   - Ability
   - Context
   - Task
   - Output
3. 不要随意改动评分维度映射、分数换算、JSON schema、前端依赖字段。
4. 修改 AI 相关代码时，必须说明：
   - 对评分准确性的影响
   - 对 JSON 结构化输出的影响
   - 对用户能力画像更新的影响
   - 对上下文处理逻辑的影响
   - 对 Prompt 分层与组装逻辑的影响
5. 修改 `ai/*`、`PromptAssembler`、`AbilityPromptBuilder`、评分/润色 Prompt 时，优先保持现有返回结构和解析流程稳定。

## 语法检查与多引擎规则

语法检查链路不是单一引擎，当前可能涉及：

- LanguageTool
- Sapling
- TextGears
- Trinka

修改语法相关逻辑时，必须评估：

1. 多引擎合并逻辑
2. 去重规则
3. span / offset 对齐
4. suppress / dismiss / fix 抑制逻辑
5. 前端依赖字段兼容性
6. 单个引擎失败时的降级路径

禁止行为：

- 只按单一引擎返回结构做破坏性调整
- 只修某个引擎字段，不检查合并结果是否仍然正确
- 删除已有错误来源、traceId、耗时、降级相关日志

## Redis 与异步任务规则

项目中 Redis 与异步任务属于核心状态层的一部分，不是可随意变更的附属逻辑。

任何新增或修改 Redis 逻辑时，必须说明：

1. key 设计
2. user / doc / task 作用域
3. TTL
4. 失效策略
5. 与现有缓存、状态恢复、轮询逻辑的兼容性

任何新增或修改评分轮询、任务状态、恢复逻辑时：

1. 优先放在 service 层或任务服务中处理
2. 不要把任务状态机散落在 controller
3. 不要把 controller 变成任务编排器
4. 需要考虑失败重试、超时、降级和历史状态恢复

## 数据库规则

数据库结构是系统核心资产，不应随意修改。

核心数据表包括：

- user_profile
- user_ability_profile
- essay_evaluation
- evaluate_task
- document

如果需要修改表结构：

1. 说明修改原因
2. 提供迁移方案
3. 确认与现有服务兼容
4. 说明对评分、画像、历史数据、任务状态的影响

不要在没有迁移说明的情况下直接改核心表结构。

## API / DTO 兼容规范

API 应遵循 REST 风格。

接口设计要求：

- 命名清晰
- 尽量避免破坏已有 API
- 返回结构化 JSON

示例：

```json
{
  "score": 82,
  "grammar_feedback": [],
  "improvement_suggestions": []
}
```

修改 `dto/writing/*`、`dto/rubric/*` 或 AI 返回 JSON 时：

1. 优先新增字段
2. 避免直接删字段或改名
3. 若必须变更，必须给兼容说明
4. 不要让前端通过猜测字段来适配后端变化
5. 任何字段调整都要检查前端解析、缓存与历史数据兼容性

## 测试要求

当后端代码发生修改时，必须先运行测试。

推荐命令：

```bash
mvn test
```

如果测试失败：

- 停止后续操作
- 输出错误信息
- 不要提交 commit

如果修改涉及评分、语法、Prompt、Redis 或异步任务，除单元测试外还应检查：

- 关键服务是否可启动
- 主要 JSON 输出结构是否未破坏
- 关键日志是否仍可用于排障

## 日志与可观测性

修改 AI 调用、评分、语法检查、任务状态相关逻辑时，应保留或补充以下信息：

- traceId
- 来源引擎 / 来源服务
- 耗时
- 失败原因
- 降级路径
- 关键输入规模（如文本长度、任务 id）

禁止为了“简化代码”删除关键链路日志。

如果确实需要减少日志，必须确认：

1. 不影响问题排查
2. 不影响 AI 调用追踪
3. 不影响评分任务状态定位

## 性能与安全

避免：

- 在 controller 中执行耗时操作
- 在请求线程中直接进行大量 AI 调用
- 在没有限流/超时/降级考虑的情况下增加外部调用

建议：

- 通过 service 层组织 AI 调用
- 在必要时采用异步处理
- 对外部 AI / 语法服务调用保留超时、重试、降级意识
- 对用户输入、Prompt 输入、日志输出做好必要的安全与脱敏考虑
